# PAAKURI 다음 개발 작업

마지막 갱신: 2026-07-21
대상 환경: Godot 4.7.1 Mono, .NET 8, Windows
현재 단계: AI 대전 유지 + ENet 호스트 권한형 P2P 프로토타입 구현 완료, 실기기 검증과 전용 서버 통합 필요

## 네트워크 통신·서버 연결·비호스트 지연 분석

### 1. 현재 사용한 네트워크 방식

현재 실제로 플레이 가능한 멀티플레이는 **Godot High-level Multiplayer API와 `ENetMultiplayerPeer`를 사용한 UDP 기반 호스트 권한형 P2P(리슨 서버) 방식**이다. 별도의 중앙 게임 서버에 두 플레이어가 접속하는 구조가 아니라, 방을 만든 플레이어의 게임 프로세스가 클라이언트와 서버 역할을 동시에 수행한다. 호스트의 Godot peer ID는 `1`이며 원격 참가자는 한 명만 접속할 수 있다.

- 호스트는 `NetworkSession.Host()`에서 `ENetMultiplayerPeer.CreateServer()`를 호출해 UDP 포트를 연다. 기본 포트는 `24872`이고, 사용 중이면 `24881`까지 순서대로 빈 포트를 찾는다.
- 참가자는 `NetworkSession.Join()`에서 `ENetMultiplayerPeer.CreateClient()`를 호출해 로비에 표시된 `IP:포트`로 접속한다.
- 접속 상태는 `PeerConnected`, `PeerDisconnected`, `ConnectedToServer`, `ConnectionFailed`, `ServerDisconnected` 이벤트로 처리한다.
- 실제 게임 데이터는 Godot C#의 `[Rpc]`와 `RpcId()`를 통해 주고받는다.
- `Master_GJ`에 별도의 헤드리스 전용 서버 프로토타입이 있지만, 현재 `Master_JS` 클라이언트의 리슨 서버 프로토콜과 포맷이 달라 직접 연결되지 않는다. 따라서 현재 플레이 중 서버 판정을 담당하는 것은 별도 전용 서버가 아니라 방을 생성한 호스트다.

호스트 권한형으로 구성했기 때문에 참가자는 자신의 위치, 체력, 탄약 또는 충돌 결과를 확정하지 않는다. 참가자는 입력만 전송하고, 호스트가 이동과 전투를 계산한 뒤 확정 상태를 참가자에게 돌려준다.

```text
참가자 입력 수집
→ UDP RPC로 호스트에 입력 전송
→ 호스트가 이동·사격·충돌·패링·피해를 계산
→ 호스트가 30Hz 스냅샷 전송
→ 참가자가 스냅샷을 화면에 보간하여 표시
```

### 2. 통신 채널과 신뢰성 설정

| 채널 | 방향 | 전송 방식 | 용도 |
| --- | --- | --- | --- |
| 입력 채널 0 | 참가자 → 호스트 | `UnreliableOrdered` | 이동, 점프, 조준, 사격, 패링 입력 |
| 스냅샷 채널 1 | 호스트 → 참가자 | `UnreliableOrdered`, 30Hz | 플레이어·총알·라운드의 확정 상태 |
| 게임 시작 채널 2 | 호스트 → 참가자 | `Reliable` | 양쪽 READY 완료 후 게임 씬 전환 |
| READY 메시지 | 양방향 | `Reliable` | 로컬/원격 준비 상태 동기화 |

`UnreliableOrdered`는 낮은 지연을 위해 패킷 재전송을 보장하지 않으며, 같은 채널에서는 최신 순서를 유지한다. 따라서 오래 도착한 패킷은 이미 더 최신 패킷을 받은 경우 버려질 수 있다. 반면 READY와 게임 시작처럼 반드시 도착해야 하는 로비 데이터는 `Reliable`로 전송한다.

### 3. 참가자가 호스트로 전달하는 데이터

참가자의 `GameManager._Process()`는 매 렌더 프레임 `Player.CaptureLocalCommand()`로 입력을 읽고 `NetworkSession.SendInput()`을 호출한다. 전송되는 `PlayerCommand`의 내용은 다음과 같다.

| 필드 | 형식 | 의미 및 검증 |
| --- | --- | --- |
| `Move` | `float` | 좌우 이동값. 호스트 수신 전에 `-1~1`로 제한 |
| `Jump` | `bool` | 해당 프레임의 점프 누름 이벤트 |
| `Down` | `bool` | 숙이기·빠른 낙하·벽 아래 이동 유지 상태 |
| `Up` | `bool` | 벽 위 이동 유지 상태 |
| `Shoot` | `bool` | 마우스 왼쪽 버튼 유지 상태 |
| `Parry` | `bool` | 해당 프레임의 패링 누름 이벤트 |
| `Aim` | `Vector2` | 조준 방향. NaN/Infinity와 영벡터를 거르고 단위 벡터로 정규화 |

호스트의 `SubmitInput()`은 등록된 원격 peer가 보낸 입력인지 확인한 다음 `RemoteInputReceived` 이벤트를 발생시킨다. `GameManager`는 이 입력을 2번 플레이어의 `_remoteCommand`에 저장하고, 호스트의 다음 물리 프레임에서 사용한다. 참가자는 위치, 속도, 체력, 탄약, 쿨타임, 총알 생성 또는 피격 결과를 호스트에 보내지 않는다.

### 4. 호스트가 참가자에게 전달하는 데이터

호스트는 `GameManager.BuildSnapshot()`으로 이진 `byte[]` 스냅샷을 만들고 초당 30회 전송한다. `BinaryWriter`로 기록되는 데이터는 다음과 같다.

1. 스냅샷 버전
2. 현재 게임 상태(`Countdown`, `Playing`, `RoundEnd`, `MatchEnd`, `Result`)
3. 현재 상태의 남은 시간
4. 선택된 맵 인덱스
5. 플레이어 1과 플레이어 2의 라운드 승수
6. 화면 메시지
7. 두 플레이어 각각의 위치, 속도, 조준 방향, 체력, 탄약, 재장전 남은 시간, 패링 쿨타임, 생존 여부
8. 현재 총알 수
9. 각 총알의 네트워크 ID, 위치, 속도, 소유 플레이어 ID

참가자의 `OnSnapshotReceived()`는 버전이 맞는 스냅샷만 읽어 맵, 점수, UI, 플레이어와 총알 복제본에 반영한다. 플레이어 위치는 매 물리 프레임 `Lerp`로 목표 위치를 따라가며, 호스트 위치와 180px 이상 차이가 나면 즉시 호스트 위치로 보정한다. 참가자 화면의 총알은 충돌 판정을 하지 않는 시각용 복제본이며 실제 충돌, 패링, 피해, 탄약, 재장전, 사망, 라운드 결과는 모두 호스트에서만 확정한다.

### 5. 비호스트에서 렉이 발생하는 이유

가장 큰 원인은 **참가자 로컬 예측(client-side prediction)이 전혀 없기 때문**이다. 참가자에서는 두 플레이어 모두 `IsNetworkReplica`로 설정되며, 자신의 캐릭터도 로컬 입력을 받아 즉시 `MoveAndSlide()`하지 않는다. 화면에 움직임이 보이려면 다음 왕복 절차가 끝나야 한다.

```text
입력 송신 대기
＋ 참가자→호스트 네트워크 지연
＋ 호스트의 다음 60Hz 물리 프레임 대기
＋ 최대 약 33.3ms의 다음 30Hz 스냅샷 대기
＋ 호스트→참가자 네트워크 지연
＋ 참가자 위치 Lerp 보간 지연
```

따라서 참가자는 대략 RTT 전체와 서버 처리·스냅샷 대기·보간 시간을 합친 지연을 체감한다. 반면 호스트는 자신의 입력을 로컬 물리 프레임에서 바로 처리하므로 같은 네트워크 왕복 과정이 없다. 이 구조적 차이 때문에 호스트는 즉각 반응하고 참가자는 무겁고 늦게 반응한다.

추가 원인은 다음과 같다.

- 게임 물리는 60Hz인데 상태 스냅샷은 30Hz이므로 참가자는 절반 빈도의 위치 표본만 받는다.
- 현재 보간은 시간순 스냅샷 버퍼나 속도 기반 외삽이 아니라, 가장 최근 목표 위치를 계속 따라가는 단순 `Lerp`다. 패킷 지터나 손실이 생기면 움직임이 늦어지거나 끊겨 보이고, 차이가 180px을 넘으면 순간이동처럼 보정된다.
- 스냅샷에 서버 tick 또는 sequence 번호가 없고 참가자에도 jitter buffer가 없어, 도착 간격이 흔들릴 때 일정한 시간축으로 재생할 수 없다.
- RTT 측정과 연결 품질 표시가 없어 실제 네트워크 지연과 보간 지연을 구분하기 어렵다.

### 6. 비호스트 입력이 씹히는 이유

입력이 씹히는 현상은 단순히 화면 반영이 늦는 경우와, 입력 패킷 자체가 실제로 소실되는 경우가 함께 존재한다.

1. **점프와 패링은 한 프레임만 `true`인 edge 입력이다.** `Input.IsActionJustPressed()`로 읽은 값은 해당 프레임의 `PlayerCommand` 한 개에만 들어간다.
2. **입력 RPC는 `UnreliableOrdered`다.** 그 한 패킷이 UDP 손실로 사라지면 재전송이나 확인 응답이 없으므로 호스트는 점프·패링 사실을 영원히 알 수 없다.
3. **최신 명령 하나만 저장한다.** 호스트의 `SetRemoteCommand()`는 입력 큐에 쌓지 않고 `_remoteCommand`를 덮어쓴다. 호스트 물리 프레임이 읽기 전에 `Jump=true` 패킷 뒤의 `Jump=false` 패킷이 도착하면, 앞의 점프 입력이 처리되지 않은 채 사라질 수 있다.
4. **늦은 입력은 ordered 처리에서 버려질 수 있다.** 네트워크 지터 때문에 점프 패킷이 늦게 도착하고 그 뒤 번호의 일반 이동 패킷이 먼저 도착하면, 늦은 점프 패킷은 오래된 패킷으로 취급될 수 있다.
5. **렌더 프레임 송신과 물리 프레임 소비 주기가 다르다.** 참가자는 가변 렌더 프레임마다 전송하지만 호스트는 60Hz 물리 프레임에서 최신 값만 한 번 읽는다. 프레임률이 높거나 지터가 있으면 여러 입력이 한 물리 프레임 사이에 겹쳐 중간 입력이 덮어써질 수 있다.
6. **짧은 사격 클릭도 같은 영향을 받을 수 있다.** 사격은 버튼 유지 상태라서 오래 누르면 다음 패킷으로 복구되지만, 아주 짧은 클릭은 `Shoot=true` 패킷 하나만 생성되어 손실 또는 덮어쓰기될 수 있다. 이동·숙이기·벽 이동은 유지 상태가 반복 전송되므로 완전히 소실될 가능성은 낮지만 다음 패킷까지 반응이 늦어진다.
7. **호스트 판정상 거부된 입력도 씹힌 것처럼 보일 수 있다.** 카운트다운, 사망, 발사 쿨타임, 재장전, 패링 준비·활성 상태, 패링 쿨타임 중에는 호스트가 행동을 실행하지 않는다. 참가자가 보는 탄약과 쿨타임 UI도 30Hz 스냅샷만큼 늦으므로 본인 화면에서는 가능해 보였던 입력이 호스트에서 거부될 수 있다.

### 7. 권장 수정 방향

- 이동에는 참가자 로컬 예측을 적용하고, 호스트 스냅샷이 오면 입력 이력으로 reconciliation한다. 사격·패링·피해 결과는 계속 호스트가 확정한다.
- `PlayerCommand`에 증가하는 sequence와 client tick을 추가하고, 호스트가 처리한 마지막 sequence를 스냅샷에 포함한다.
- 점프·패링·짧은 사격 같은 edge 입력은 입력 큐에 보관하고 호스트 확인 전까지 여러 입력 패킷에 중복 포함하거나, 별도의 신뢰 가능한 이벤트/확인 응답 방식을 사용한다. 모든 이동 입력을 무조건 `Reliable`로 바꾸면 오래된 패킷 재전송 때문에 head-of-line 지연이 생길 수 있으므로 상태 입력과 edge 이벤트를 구분하는 편이 적절하다.
- 호스트는 받은 최신 명령 하나만 보관하지 말고 sequence 순서의 입력 큐를 물리 tick 단위로 소비한다.
- 입력 송신률을 고정 tick에 맞추고, 스냅샷에 server tick을 넣어 오래된 스냅샷을 명시적으로 무시한다.
- 참가자 렌더링에 짧은 jitter buffer와 시간 기준 보간을 사용하고, 필요한 경우 속도로 짧게 외삽한다.
- 개발 화면에 RTT, 패킷 손실률, 마지막 입력 sequence, 서버 tick, 보정 거리 값을 표시해 실제 원인을 측정한다.

## 절대 유지할 조건

- AI 대전 모드를 삭제하지 않는다.
- 멀티플레이의 위치, 총알, 충돌, 패링, 피해, 사망, 점수는 서버/호스트만 확정한다.
- 클라이언트가 보낸 위치, 체력, 탄약, 쿨타임 값은 신뢰하지 않는다.
- 사용자 작업인 `hell` 및 PNG import 파일 삭제 상태를 되돌리거나 커밋에 임의로 포함하지 않는다.
- 클라이언트 저장소는 `Master_JS`, 분리된 헤드리스 서버 저장소는 `Master_GJ`다.
- 현재 완성 맵 리소스는 `map_a.png`, `map_c.png`, `map_d.png`, `map_g.png` 네 개뿐이다.

- [ ] 호스트가 오래되거나 중복된 jump/parry edge 패킷을 버리도록 한다.
- [ ] 스냅샷에 서버 tick을 추가하고 클라이언트가 오래된 스냅샷을 무시하도록 한다.
- [ ] 현재 30Hz 보간을 jitter buffer 기반으로 변경하고 순간이동 임계값 180px을 지연 테스트 결과에 맞춰 조정한다.
- [ ] 참가자 로컬 플레이어에 이동 예측/서버 reconciliation을 추가하되 사격·패링·피해는 예측하지 않는다.
- [ ] 잘못된 payload 크기, NaN/Infinity 조준, 과도한 입력 빈도에 대한 rate limit과 연결 종료 정책을 추가한다.

완료 조건: 150ms RTT에서 로컬 이동이 조작 가능하고 5% 손실 후 위치가 서버 상태로 복구되며 중복 점프·중복 패링이 발생하지 않아야 한다.

### SERVER-001 `Master_GJ` 전용 서버 프로토콜 통합

- [ ] 현재 `Master_GJ`의 2클라이언트 전용 서버 스냅샷과 `Master_JS` 리슨 서버 스냅샷 포맷이 다르므로 하나의 버전 명세로 통합한다.

- [ ] `Master_JS` 로비에 `전용 서버 참가` 모드를 추가하고 peer slot 배정을 서버에서 받는다.
- [ ] `Master_GJ`의 임시 사각형 바닥 판정을 네 맵의 실제 충돌 데이터와 같은 서버용 정적 데이터로 교체한다.
- [ ] 벽 붙기/오르기/미끄러지기/벽 점프/빠른 낙하/숙이기 판정을 전용 서버 시뮬레이션에 이식한다.
- [ ] 피격 무적 0.15초, 패링 준비 0.05초/활성 0.15초/후딜 0.25초를 전용 서버에서 정확히 구현한다.
