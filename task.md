# PAAKURI 다음 개발 작업

마지막 갱신: 2026-07-21
대상 환경: Godot 4.7.1 Mono, .NET 8, Windows
현재 단계: AI 대전 유지 + ENet 호스트 권한형 P2P 프로토타입 구현 완료, 실기기 검증과 전용 서버 통합 필요

## 절대 유지할 조건

- AI 대전 모드를 삭제하지 않는다.
- 멀티플레이의 위치, 총알, 충돌, 패링, 피해, 사망, 점수는 서버/호스트만 확정한다.
- 클라이언트가 보낸 위치, 체력, 탄약, 쿨타임 값은 신뢰하지 않는다.
- 사용자 작업인 `hell` 및 PNG import 파일 삭제 상태를 되돌리거나 커밋에 임의로 포함하지 않는다.
- 클라이언트 저장소는 `Master_JS`, 분리된 헤드리스 서버 저장소는 `Master_GJ`다.
- 현재 완성 맵 리소스는 `map_a.png`, `map_c.png`, `map_d.png`, `map_g.png` 네 개뿐이다.

## P0 — 다음 에이전트가 가장 먼저 할 작업

### NET-QA-001 두 PC 실기기 멀티플레이 검증

- [ ] 같은 LAN의 Windows PC 두 대에서 호스트가 `멀티 방 생성`, 참가자가 `호스트LAN주소:24872`로 접속한다.
- [ ] 양쪽 READY 후 같은 맵, 같은 스폰, 같은 카운트다운이 표시되는지 확인한다.
- [ ] 참가자 이동/점프/숙이기/벽 이동/조준/사격/패링 입력이 호스트 판정에 반영되는지 확인한다.
- [ ] 총알 위치·소유자, 패링 반전, 체력·탄약·재장전·패링 쿨타임, 라운드 점수가 양쪽에서 일치하는지 확인한다.
- [ ] 50/100/150ms 지연과 1/3/5% 패킷 손실에서 위치 보간, 점프 edge 입력, 패링 판정이 깨지지 않는지 측정한다.
- [ ] 호스트 종료와 참가자 종료 각각에서 남은 플레이어가 타이틀로 복귀하고 이유 문구를 보는지 확인한다.
- [ ] 호스트 결과 화면의 재대결이 참가자 화면도 새 카운트다운으로 되돌리는지 확인한다.

완료 조건: 두 PC에서 3판 2선승 경기를 5회 완료하고 관리 코드 예외, RPC 오류, 점수 불일치가 없어야 한다. 평균/최대 RTT와 발견한 보정 수치를 `work.md`에 기록한다.

### NET-002 입력 시퀀스와 스냅샷 강건성

- [ ] `PlayerCommand`에 증가하는 입력 시퀀스 번호와 클라이언트 tick을 추가한다.
- [ ] 호스트가 오래되거나 중복된 jump/parry edge 패킷을 버리도록 한다.
- [ ] 스냅샷에 서버 tick을 추가하고 클라이언트가 오래된 스냅샷을 무시하도록 한다.
- [ ] 현재 30Hz 보간을 jitter buffer 기반으로 변경하고 순간이동 임계값 180px을 지연 테스트 결과에 맞춰 조정한다.
- [ ] 참가자 로컬 플레이어에 이동 예측/서버 reconciliation을 추가하되 사격·패링·피해는 예측하지 않는다.
- [ ] 잘못된 payload 크기, NaN/Infinity 조준, 과도한 입력 빈도에 대한 rate limit과 연결 종료 정책을 추가한다.

완료 조건: 150ms RTT에서 로컬 이동이 조작 가능하고 5% 손실 후 위치가 서버 상태로 복구되며 중복 점프·중복 패링이 발생하지 않아야 한다.

### SERVER-001 `Master_GJ` 전용 서버 프로토콜 통합

- [ ] 현재 `Master_GJ`의 2클라이언트 전용 서버 스냅샷과 `Master_JS` 리슨 서버 스냅샷 포맷이 다르므로 하나의 버전 명세로 통합한다.
- [ ] 공용 프로토콜 DTO/직렬화 코드를 별도 .NET 프로젝트 또는 동기화 가능한 파일로 분리한다.
- [ ] `Master_JS` 로비에 `전용 서버 참가` 모드를 추가하고 peer slot 배정을 서버에서 받는다.
- [ ] `Master_GJ`의 임시 사각형 바닥 판정을 네 맵의 실제 충돌 데이터와 같은 서버용 정적 데이터로 교체한다.
- [ ] 벽 붙기/오르기/미끄러지기/벽 점프/빠른 낙하/숙이기 판정을 전용 서버 시뮬레이션에 이식한다.
- [ ] 피격 무적 0.15초, 패링 준비 0.05초/활성 0.15초/후딜 0.25초를 전용 서버에서 정확히 구현한다.
- [ ] 두 클라이언트 연결, READY, 3판 2선승, 재대결, 한 명 연결 종료를 자동 통합 테스트한다.

주의: 현재 실제 플레이 가능한 멀티는 `Master_JS`의 플레이어 호스트 리슨 서버다. `Master_GJ`는 분리된 헤드리스 서버 기준 구현이지만 아직 `Master_JS` 전용 서버 참가 모드와 직접 호환되지 않는다.

## P1 — 인터넷 방과 로비 완성

### LOBBY-001 방 코드/인터넷 연결

- [ ] 현재 방 코드는 `IP:UDP포트` 형식이다. 랜덤 짧은 코드가 필요하면 외부 rendezvous/매치메이킹 서비스를 설계한다.
- [ ] ENet은 UDP를 사용하므로 UDP 24872 포트 포워딩 안내를 UI에 추가한다.
- [ ] Godot `UPNP` 자동 포트 매핑을 선택 기능으로 추가하고 실패 이유를 표시한다.
- [ ] CGNAT/대칭 NAT 사용자를 위해 릴레이 서버 또는 Steam/EOS/WebRTC 기반 NAT traversal을 비교 검증한다.
- [ ] 방 비밀번호/세션 nonce, 참가 승인, 준비 취소, 중복 접속 거부를 추가한다.
- [ ] 공인 인터넷 공개 전 RPC 인증, rate limit, protocol version 거부를 구현한다.

### LOBBY-002 재대결과 상태 복구

- [ ] 결과 화면에서 양쪽이 재대결 READY를 눌러야 새 경기가 시작되도록 한다.
- [ ] 참가자도 재대결 의사를 보낼 수 있게 하고 현재 비활성화된 참가자 RESTART 버튼을 교체한다.
- [ ] 씬 전환 중 연결 종료, READY 직후 종료, 라운드 종료 직후 종료 회귀 테스트를 추가한다.
- [ ] 호스트 마이그레이션은 이번 프로토타입 범위 밖으로 명시하거나 별도 설계한다.

## 매 작업 후 검증 명령

1. 클라이언트: `dotnet build Master_JS.csproj --nologo`
2. 클라이언트 임포트: Godot 4.7.1 Mono `--headless --path . --editor --quit`
3. AI 스모크: `--headless --path . --scene res://Scene/main.tscn --fixed-fps 60 --quit-after 600`
4. 호스트/클라이언트 2프로세스: `-- --paakuri-host --paakuri-auto-ready` / `-- --paakuri-join=127.0.0.1:24872 --paakuri-auto-ready`
5. 서버: `dotnet build Master_GJ.csproj --nologo`
6. 서버 스모크: `--headless --path . --quit-after 120 -- --port=24873`
7. 양쪽 저장소: `git diff --check`
